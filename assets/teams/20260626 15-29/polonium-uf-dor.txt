Task description
Every Task should be well described:
- What has to be done
- Has acceptance criteria and should be related to Epic Acceptance Criteria
- UX design (if applicable) should be represent actual workflow and in form of Figma (link)
- Every Tasks should be connected to Epic
- Dependencies (if applicable) in description
- Dependences/Relationships between tasks should be visible in Jira (if applicable)

Estimates Provided
The development team has provided an estimate for the work, or the ticket has been broken down to a size that allows for reasonable estimation.

Work size
Scope is small enough to fit within a sprint

Additional criteria:
- We have clear Acceptance Criteria
- We have defined testcases
- In case of UI changes, there are mockups provided by UX/UI team
- Task is estimated by the team
- Task was investigated by engineering team - It needs to have linked task which is clearly created as Preparation or Investigation
- We have list of involved teams and Engineers/PMs in case of bigger projects
- All dependencies like other team's tasks and blockers are resolved
- All risks are estimated and listed
- In case of change of requirements new task should be provided. We don't want to change scope during development
- Defined ways of success measurements. Statistics and other ways to measure difference before and after
- If task is related to documentation there is necessity to define templates or common look of document with stakeholders
