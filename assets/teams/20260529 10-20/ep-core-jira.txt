## Context

EP Core operates a growing set of workloads with very different trust, blast-radius, and availability requirements. Currently there is no written record where we should migrate our workload to.

This ticket is to produce an **ADR** that captures the decision and the reasoning behind it so we would know which accounts are for what purpose, and how to use them. This will allow us to schedule the EP Core migration better.

**Topics the ADR must address**

1\. Where to place typical EP Core and EP Data workloads

2\. Sensitive / privileged resources

* **Terraform Enterprise (TFE)** — has AWS credentials and state for all accounts; should it be isolated
* **Vault** — secrets backend; blast radius of a compromise is very high
* Should these run on a dedicated account/cluster? What are the access control implications?

3\. **CI/CD** — self-hosted GitHub Actions runners

4\. **Observability stack**

* VictoriaMetrics, Grafana, VictoriaMetrics, Loki — need to scrape metrics from all clusters
* **Should observability run in a shared account/cluster accessible from all environments (network segement)?**

5\. **AWS account structure**

* Should each concern (platform, data, CICD, observability) get a dedicated AWS account?
* What are the pros and cons vs. fewer, larger accounts?

## Requirements

* We MUST create ADR with the description which **AWS account** EP should provision for EP
* We MUST create ADR with the description which **EKS accounts** EP should provision for EP
* We MUST explicitly tell which Global Network segment given EKS will be using
* We MUST list pros and cons of every choice
* We MUST give it to Security for review and approval
* We MUST decide as a team on the approach

Summary: Issue in resource pool export_rp 
Assignee: Loy Noronha
URL: https://adgear.atlassian.net/browse/EPCW-86
Created: 2026-05-20T02:15:56.573-0400
Updated: 2026-05-26T08:47:43.898-0400
================================================================================

Description:
## 📝 Context & Background

Following our migration from Vertica 10 to Vertica 24.4 running on Kubernetes Eon Mode, we encountered a critical performance regression where specific data export queries (running via the raven_export user out of the export_rp pool) were stalling for 111+ seconds.  
Investigation revealed that the Vertica 24.4 query optimizer introduces stricter execution rules for unsegmented temporary tables (temp_0000) and standard UNION clauses. This layout forced a global network broadcast re-segmentation, flooding our 45-node Kubernetes cluster overlay network with \~90 GB of distributed traffic (\~2 GB sent/received per pod). Because export_rp was configured with a MAXMEMORYSIZE of 90% and a high PLANNEDCONCURRENCY of 5, individual queries were being thread-starved while simultaneously endangering the core general pool's memory stability.

## 🎯 Objectives

* Isolate Resource Footprint: Prevent the export_rp pool from dynamically scaling to 90%, which starves the core general execution pool.
* Maximize Thread Performance: Allocate greater internal pipeline thread budgets to heavy network-bound queries.
* Enforce Safety Limits: Protect the 45 Kubernetes statefulset pods from memory-exhaustion or long application hangs.

---

## 🚀 Goals

* Reduce target query execution time from 111 seconds to under 15 seconds without modifying application source code or table schemas.
* Safeguard cluster-wide memory limits from dynamic overcommitment during high-volume application runs.

---

## 📝 Acceptance Criteria

* export_rp pool properties successfully altered across all 45 nodes.
* Maximum memory threshold clamped to prevent cluster memory starvation.
* Initial execution thread budget per query expanded by decreasing planned concurrency slots.
* Single-query memory ceiling implemented to prevent any single bad execution path from overwhelming a pod.
* Session/query timeout duration aligned to match standard cluster timeouts (5 minutes).
* Verified query execution duration for raven_export users consistently executes in $\\le$ 10 seconds.

---

## 🛠️ Implementation Steps

Run the following administration overrides in the database console to update the cluster topology:

\-- Step 1: Clamp maximum resource pool expansion to protect the 'general' poolALTER RESOURCE POOL export_rp MAXMEMORYSIZE '40%';  
\-- Step 2: Drop concurrency slots to force larger worker-thread allocations per queryALTER RESOURCE POOL export_rp PLANNEDCONCURRENCY 2;  
\-- Step 3: Inject a per-query memory ceiling to block dynamic runaway allocationALTER RESOURCE POOL export_rp MAXQUERYMEMORYSIZE '15G';  
\-- Step 4: Align queue timeouts to match cluster standards and prevent app hangsALTER RESOURCE POOL export_rp QUEUETIMEOUT '00:05';

---

## 🏆 Measurable Outcomes

* Performance Velocity: Target query step execution dropped from 111.9 seconds to 10 seconds (a \~91% processing speed improvement).
* Parallel Execution Safety: Thread budget optimization successfully managed the \~2 GB per-node network broadcast queue efficiently within container limits.
* Cluster Reliability: The database engine is fully insulated against out-of-memory crashes or service degradation during concurrent ETL and export schedules.

---

Description:
## Context

We have decided in [this slack thread](https://github.com/adgear/k8s-core) that EP Data should start using EP Core ArgoCD, and be able to deploy their applications using <custom data-type="smartlink" data-id="id-0">https://github.com/adgear/k8s-core</custom> till now they used <custom data-type="smartlink" data-id="id-1">https://github.com/adgear/ep-tenant-operators/</custom> .

But we need to put some safeguards on it. We should separate EP Core apps from EP Data ones.

EP Data team  now has limited, hardcoded per-app read/sync access via `role:cg-ep-data-readsync`:

```
g, A.ArgoCD.cg-ep-data.Admin, role:cg-ep-data-readsync
p, role:cg-ep-data-readsync, applications, get, k8s-core/kafka-topics, allow
p, role:cg-ep-data-readsync, applications, sync, k8s-core/kafka-topics, allow
...
```

This approach does not scale — every new app they own requires new RBAC lines maintained by EP Core. The EP Data team should be self-sufficient for their own applications.

## Requirements

* We MUST create a separate ArgoCD project `cg-ep-data` for EP Data
* We MUST give EP Data team **full admin** access to their project `cg-ep-data`
* We MUST give them read only access to EP Core applications
* We MUST move all existing EP Data apps (like kafka-topics, maybe more) to project `cg-ep-data`, but only if this is **NON-DESTRUCTIVE**

    * We MUST not touch tenant operators
    

## Implementation Plan

Below is a suggestion

Created using Claude, verify the plan on your own.

### 1. In <custom data-type="smartlink" data-id="id-2">https://github.com/adgear/k8s-core</custom> create `templates/project.cg-ep-data.yaml`

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: cg-ep-data
spec:
  clusterResourceWhitelist:
  - group: '*'
    kind: '*'
  destinations:
  {{- range $ns, $enabled := .Values.cgEpDataDestinationNamespaces }}
  {{- if $enabled }}
  - namespace: {{$ns}}
    server: https://kubernetes.default.svc
  {{- end }}
  {{- end }}
  sourceRepos:
  {{- $repos := list }}
  {{- range $key, $value := .Values.cgEpDataComponents }}
  {{- if and $value.enabled $value.repoUrl }}
    {{- $repos = append $repos $value.repoUrl }}
  {{- end }}
  {{- end }}
  {{- range ( sortAlpha $repos | uniq ) }}
  - {{ . | squote }}
  {{- end }}
status: {}
```

### 2. Add `cgEpDataDestinationNamespaces` and `cgEpDataComponents` to `values.yaml`

```yaml
# Destination namespaces allowed in the cg-ep-data AppProject
cgEpDataDestinationNamespaces: {}

# Components owned by the EP Data team (used to build AppProject sourceRepos)
cgEpDataComponents:
  kafka-topics:
    enabled: false
    repoUrl: 'https://github.com/adgear/kafka-topics.git'
```

Per-cluster values files override as needed, e.g. in a Rancher cluster where kafka-topics is active:

```yaml
cgEpDataDestinationNamespaces:
  ep-kafka-use1-rprod: true

cgEpDataComponents:
  kafka-topics:
    enabled: true
```

### 3. Move `kafka-topics` app to the new project

In `templates/app.kafka-topics.yaml`, change:

```yaml
project: k8s-core  →  project: cg-ep-data
```

> **Safe and non-destructive.** Changing `spec.project` on an existing ArgoCD Application only updates the RBAC scope — it does not delete, recreate, or affect the application or its managed Kubernetes resources.

Both changes land in the same PR and sync together. ArgoCD processes AppProject resources before Applications, so the project will exist before the app references it.

Test if this is in fact non destructive

Maybe there are other EP Data apps, move them as well.

### 4. Update RBAC in `values.yaml`

```yaml
rbac:
  policy.csv: |
    # existing
    g, A.ArgoCD.Admin, role:admin
    g, A.ArgoCD.Engineering, role:readonly
    g, A.ArgoCD.Security, role:readonly
    g, A.ArgoCD.SRE, role:sre

    # EP Data: admin on their project + read/sync on platform apps
    g, A.ArgoCD.cg-ep-data.Admin, role:cg-ep-data-admin   # NEW
    g, A.ArgoCD.cg-ep-data.Admin, role:cg-ep-data-readsync # keep existing

    # new role: full control on cg-ep-data project
    p, role:cg-ep-data-admin, applications, *, cg-ep-data/*, allow
    p, role:cg-ep-data-admin, logs, get, cg-ep-data/*, allow
    p, role:cg-ep-data-admin, repositories, get, *, allow
    p, role:cg-ep-data-admin, clusters, get, *, allow
    p, role:cg-ep-data-admin, projects, get, cg-ep-data, allow

    # existing readsync role (unchanged)
    p, role:cg-ep-data-readsync, applications, get, k8s-core/kafka-topics, allow
    ...
```

## Affected Files

* `k8s-core/templates/project.cg-ep-data.yaml` — new file
* `k8s-core/templates/app.kafka-topics.yaml` — change `project:` field
* `k8s-core/values.yaml` — add `cgEpDataDestinationNamespaces`, `cgEpDataComponents`, update RBAC `policy.csv`
* Per-cluster values files — add `cgEpDataDestinationNamespaces` and enable `cgEpDataComponents` where applicable

Description:
## Context

Recently when we have introduced an incident change <custom data-type="smartlink" data-id="id-0">https://github.com/adgear/kruise-jobs/pull/139</custom> in <custom data-type="smartlink" data-id="id-1">https://github.com/adgear/kruise-jobs</custom>, which modifies `.github/workflows/sonarqube.yml` we have triggered a massive rollout of all kruise jobs to all clusters at once. It stressed `etcd`on some servers and triggered multiple alerts.

When diagnosing this issue we have learned that we trigger kruise jobs rollout on every commit to the repository (regardless if this is about kruise jobs or not):

```
apiVersion: apps.kruise.io/v1alpha1
kind: BroadcastJob
metadata:
  name: check-for-malware-{{ include "kruise-jobs.gitSHA" . }} 
```

To avoid any further damage we have pinned kruise-jobs to a specific sha: [https://github.com/adgear/kruise-jobs/pull/140/changes](https://github.com/adgear/kruise-jobs/pull/140/changes) and disable auto-sync in <custom data-type="smartlink" data-id="id-2">https://github.com/adgear/k8s-core/pull/1332</custom> .

We need to improve this mechanism and unblock release of kruise jobs again.

---

## Requirements

* We MUST review all kruise jobs and remove the ones that we think are no longer used. We MUST present it to the team.
* There are some EP Data jobs, ask EP Data if they are still needed.
* We MUST trigger kruise jobs mechanism only when a kruise job specific change is made:

    * For example we could create a git tag per kruise-job, and start versioning it. Pin kruise-job `name` to use specific tag. This way we would be able to target the release to a specific kruise job.
    
* We MUST optimize kruise jobs, for example:

    * <custom data-type="smartlink" data-id="id-3">https://github.com/adgear/kruise-jobs/blob/master/templates/disable-snapd.yaml</custom><custom data-type="smartlink" data-id="id-4">https://github.com/adgear/kruise-jobs/blob/master/templates/disable-unattended-os-upgrades.yaml</custom> most probably could be merged?
    * <custom data-type="smartlink" data-id="id-5">https://github.com/adgear/kruise-jobs/blob/master/templates/root-pass-rotation.yaml</custom>  is this still needed? Do we have root password at all?
    
* When we improve kruise-jobs we MUST re-enable auto sync
