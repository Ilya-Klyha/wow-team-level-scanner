Assignee: Katarzyna Kokieć

Description:
**Update 26.05 – Changed requirements: Log emission conditions**

The implementation emits 3 log events (instead of the original 2). Conditions for each:

| Log Event | Fires when |
| --- | --- |
| `GenAiReportViewed` | User data is ready (`userId`, `reportId`, `countryCode` are available). Fires once per report view, regardless of summary presence. |
| `GenAiSummaryViewed` | All of: user data is ready + the summary section is at least 5% visible in the viewport (partial visibility). Fires once per report view. |
| `GenAiSummaryFullyViewed` | All of: user data is ready + the summary section is at least 90% visible in the viewport + summary content is fully loaded (`isSummaryReady`). Fires once per report view. |

  
  
**Summary**

Emit Grafana Faro log events when the GenAI summary section is displayed to the user. This data feeds the success metric: "% of Pre-Campaign Insights where summary viewed ≥80% (3 months)" defined in the GenAI Summary tech spec.

### Context

* The GenAI summary section is shown on the Pre-Campaign Insight report detail page when a generated summary exists (`GET /reports/{id}/genai/summary` returns content)
* The frontend already uses `@grafana/faro-web-sdk` (v1.2.3) with a reusable `useGrafanaLogs` hook (`src/hooks/use-grafana-logs.ts`) that wraps `faro.api.pushLog()`
* Existing usage pattern: `pushLog` with `LogLevel.LOG`, a `type` field in context, and relevant metadata (see `src/pages/campaigns/hooks/logs.ts` for reference)
* The summary section is gated behind a feature flag — metrics should only fire when the feature is enabled and the section is actually rendered

### Flow

1. User opens a Pre-Campaign Insight report that has status `COMPLETE_MEDIA`
2. Emit `GenAiReportViewed` event (denominator for adoption metric)
3. Frontend calls `GET /reports/{id}/genai/summary`
4. If summary content exists, the summary section renders
5. On section render (visible in viewport): emit `GenAiSummaryViewed` log event (numerator)
6. Grafana dashboard computes adoption rate: `GenAiSummaryViewed / GenAiReportViewed`

### Scope

* Create a dedicated hook (e.g., `useGenAiSummaryLogs`) following the pattern of `useCampaignActionGrafanaLogs`
* Emit `GenAiReportViewed` event with context: `{ type: "GenAiReportViewed", reportId, reportVersion, country, buyerId, userId }` — fires on every completed report view regardless of summary presence
* Emit `GenAiSummaryViewed` event with context: `{ type: "GenAiSummaryViewed", reportId, reportVersion, country, buyerId, userId }` — fires only when summary content is rendered
* Emit each event once per report view (deduplicate if component re-renders without user navigation)
* Only emit when feature flag is active
* Create a Grafana dashboard panel visualizing summary adoption rate (`GenAiSummaryViewed / GenAiReportViewed`)

### Out of Scope

* Chat interaction metrics (separate task)
* Feedback/rating tracking (separate task)
* Backend changes

### Dependencies

* GenAI summary GET endpoint (RSW-903)

### Definition of Done

- [ ] `GenAiReportViewed` event emitted exactly once when user opens a completed Pre-Campaign Insight report
- [ ] `GenAiSummaryViewed` event emitted exactly once when the summary section renders with content
- [ ] Events include `reportId`, `reportVersion`, `country`, `buyerId`, `userId` in context
- [ ] No events emitted when feature flag is disabled
- [ ] Grafana dashboard panel created showing summary adoption rate (target: ≥80%)
- [ ] Unit tests for the hook: emission logic and deduplication

### Acceptance Criteria

**Conversation State Machine** (`conversation.go`)

* Client struct wrapping `bedrockruntime.Client` with model, max tokens (default 4096), temperature (0.1), request timeout, stream timeout, and optional throttle checker
* Custom HTTP transport: `MaxIdleConns=100`, `MaxIdleConnsPerHost=10`, `IdleConnTimeout=90s` for reduced TTFT

* Partial result return on timeout or cancellation instead of error (set `Interrupted=true`)
* Three prompt cache points: after system prompt, after last message, after tool specs — enabling 90% cost reduction on cache hits

* Loop up to `maxToolIterations=10` on `stopReason == tool_use`
* Parallel tool execution: launch concurrent goroutines per tool call, each with per-tool timeout context
* Panic recovery in each goroutine with `recovered()` guard
* Buffered result channel to prevent goroutine leaks
* On context cancellation mid-execution: drain buffered results, return partial text + accumulated tokens
* Per-tool timeouts: browse_knowledge 10s, generate_email 180s, review_email 150s, create_reviewed_email 330s

**Tool Definitions** (`tools.go`)

* `get_report` — fetch formatted report sections with `section`, `sections`, `exclude`, `include_sources` parameters; max 12,000 char result with truncation hint

* `generate_email` — single Sonnet call for email generation with `tone` and `focus` parameters
* `review_email` — Opus multi-phase email review with `email_text` parameter
* `create_reviewed_email` — full generate → review → refine pipeline with `tone` and `focus` parameters

* Multi-part queries with 2+ "and" conjunctions where at least one segment >4 words → Sonnet
* Context-aware continuation: short follow-up to Sonnet conversation (last assistant response >1000 chars) → maintain Sonnet

**Prompt Construction** (`prompts.go`)

* Prompts struct caching base system prompt, review system prompt, review rubric, merge template, glue template, 4 dimension rubrics, and generation rules loaded from knowledge files
* Length constraint primacy: word budget stated first (before report data) and reinforced at end of generation prompts
* Total View adaptation: inject constraint forbidding unexposed-audience narrative, adjust completeness rubric to 2-pillar evaluation
* Section-parallel prompt building for 3 groups: framing (sections 1-2, 250w cap), analysis (sections 3-4, 270w cap), samsung (sections 5-6, 280w cap)
* Data slicing per section group: framing gets campaign + key KPIs, analysis gets key KPIs + streaming + overlaps, samsung gets key KPIs + Samsung TV Plus

**Event Streaming System**

* Event channel (`chan<- Event`) for real-time UI feedback
* 20+ event types: text delta, tool start/input/result, review phase/score, section delta (with order for positioning), generation draft/final, token usage, error, interrupted, clarification, review summary, cache telemetry, message queued/consumed/cancelled
* Safe emit with panic recovery for closed channels, nil channel guard, non-blocking

**Knowledge Browser** (`browser.go`)

* Query formats: `"path/to/file.md"` (full file), `"path/ ##"` (list headers), `"path/ ## keyword"` (search sections), `"path/file.md ## keyword"` (search in specific file)

**Conversation Lifecycle** (`manager/manager.go`)
